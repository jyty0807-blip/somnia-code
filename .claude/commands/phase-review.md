Phase 구현 완료 후 자동 검증 스킬. 개발 + 디자인 리뷰를 병렬 실행하고, 빌드 검증까지 수행한 뒤 결과를 보고한다.

## 실행 절차

### 1단계: 변경 파일 파악
`git diff --name-only HEAD` 또는 $ARGUMENTS에서 이번 Phase에서 변경된 파일 목록을 파악한다.

### 2단계: 병렬 리뷰 (3개 동시 실행)

아래 3개를 **반드시 병렬로** 실행한다:

**A. 개발 리뷰 에이전트** — Agent(model: haiku)로 스폰:
```
/dev-review 의 전체 지침을 그대로 사용하되, 리뷰 대상을 이번 Phase 변경 파일로 지정.
변경된 파일들을 모두 읽고 dev-review 체크리스트(중복, 재사용성, 용량, 아키텍처, 미사용 코드)로 검수.
```

**B. 디자인 리뷰 에이전트** — Agent(model: haiku)로 스폰:
```
/ui-review 의 전체 지침을 그대로 사용하되, 리뷰 대상을 이번 Phase에서 UI가 변경된 .jsx 파일로 지정.
변경된 JSX 파일들을 읽고 ui-review 체크리스트(간격, 정렬, 텍스트 크기/두께, 줄바꿈, 미니멀 원칙)로 검수.
```

**C. 빌드 검증** — Bash로 `npx vite build` 실행.

### 3단계: 결과 종합 보고

3개 결과를 아래 형식으로 종합하여 사용자에게 보고한다:

```
## Phase N 검증 결과

### 개발 리뷰
(에이전트 결과 요약 — 🔴/🟡/🟢 건수 + 주요 지적)

### 디자인 리뷰
(에이전트 결과 요약 — 🔴/🟡/🟢 건수 + 주요 지적)

### 빌드 검증
(성공/실패 + 번들 크기)

### 판정
- 🔴 N건 → 수정 필요
- 🟡 N건 → 권장 수정
- 🟢 N건 → 양호

승인 대기 중입니다. 수정 진행할까요, 또는 승인하시겠습니까?
```

## 중요 규칙
- 이 스킬 실행 후 **절대로** 다음 Phase 작업을 시작하지 않는다.
- 리뷰 결과에 🔴이 있어도 자동 수정하지 않는다 — 결과만 보고하고 사용자 판단을 기다린다.
- PROJECT-STATUS.md, CHANGELOG.md를 직접 수정하지 않는다.
- **사용자가 승인하면 반드시 `/phase-approve`를 호출하여 진행도와 변경사항을 업데이트한다.** 수동 업데이트 금지.

## 강제 사이클 (위반 시 사용자에게 경고됨)
```
구현 완료 → /phase-review (이 스킬)
    → 결과 보고 → 사용자 승인 대기
    → 승인 시 → /phase-approve (진행도 + CHANGELOG 자동 업데이트)
    → /phase-approve 완료 후에만 다음 작업 가능
```
